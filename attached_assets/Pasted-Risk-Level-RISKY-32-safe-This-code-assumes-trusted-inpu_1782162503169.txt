Suggestion Cards
Hardcoded JWT secret
Why it matters:
Anyone who learns "mysecretkey" can mint valid tokens and impersonate users.
Fix:
Load the secret from process.env.JWT_SECRET, require a strong length, and never commit
it to source control.
Unsafe Authorization parsing
Why it matters:
req.headers.authorization can be undefined or malformed, and auth.split(' ')[1] will throw before returning a 401.
Fix:
Validate the header format explicitly and reject anything that is not exactly "Bearer <token>".
Unhandled JWT verification errors
Why it matters:
jwt.verify throws on expired or invalid tokens, turning normal auth failures into server errors.
Fix:
Wrap jwt.verify in try/catch and return 401 for TokenExpiredError and JsonWebTokenError.
No input validation
Why it matters:
Missing or invalid username/password values can create broken accounts or trigger runtime errors in bcrypt.
Fix:
Check req.body, enforce string types, trim username, and apply length limits before hashing or comparing passwords.
Duplicate usernames allowed
Why it matters:
Multiple records with the same username make authentication ambiguous and can lock users out or create account confusion.
Fix:
Reject registration when users.find finds an existing normalized username and return HTTP 409.

In-memory user storage
Why it matters:
All accounts disappear on restart and the app cannot scale across multiple instances.
Fix:
Use a persistent database with a unique index on username.
Identity tied only to username
Why it matters:
If username changes or is not unique, tokens no longer map reliably to the intended account.
Fix:
Store a stable user id in the token payload and look up the user by that id.
